# 태그 기능 현황 분석 및 개선 계획

## 1. 현재 구조 요약

### 1.1 DB
- **tag**: `id`, `name`, `color` (name UNIQUE)
- **problem_tag**: `problem_id`, `tag_id` (다대다 중간 테이블)
- `data.sql`에 샘플 태그 1개: id=1, name='기본'

### 1.2 백엔드
- **태그 CRUD**: `TagController` → `TagService` (생성, 목록, 단건 조회)
- **문제–태그 연결**
  - `ProblemCreateRequest` / `ProblemUpdateRequest`: `tagIds`(선택), **`tagNames`(선택, 우선)**
  - `tagNames`가 있으면 **이름으로 찾거나 없으면 새로 생성** (`TagService.findOrCreateByName`)
  - 즉, **사용자가 입력한 문자열이 그대로 새 태그로 등록 가능** (자유 태깅)
- **태그별 검색**: `GET /api/problems?tagId=...` → `ProblemRepository.findDistinctByTags_Id(tagId)` 사용

### 1.3 프론트엔드
- **문제 등록** (`CreateProblemSteps`): 태그를 **자유 입력** (문자열 배열 `formData.tags`) → API에 `tagNames`로 전달
- **태그별 검색** (`ProblemsByUnit`): `tagApi.list()`로 태그 목록 조회 후 `tagId` 선택 → `problemApi.getByTag(tagId)` 호출
- **표시** (`ProblemCard`): `problem.tags`(이름 배열)만 사용해 태그 뱃지 표시

## 2. 현재 문제점

### 2.1 문제 등록 시 태그가 표시되지 않는 현상
- **가능 원인**
  1. **등록 직후 화면**: 등록 성공 후 보여주는 화면(예: 완료 단계)에서 생성된 문제의 `tags`를 노출하지 않거나, 단순히 `resultId`만 보여주고 있을 수 있음.
  2. **목록 진입 경로**: 등록 후 “문제 목록”으로 가서 **태그별 검색**으로 들어가면, 아래 2.2 때문에 **해당 태그 하나만** 보이고, 방금 넣은 다른 태그들은 안 보일 수 있음.
  3. **multipart 파싱**: `POST /api/problems`가 `multipart/form-data`이고 `request` 파트가 JSON인데, 일부 환경에서 `tagNames` 필드가 누락되거나 빈 배열로 파싱될 가능성 (가능하면 로그로 한 번 확인하는 것이 좋음).

백엔드 로직 자체는 `tagNames`가 오면 `findOrCreateByName`으로 태그를 만들고 `problem.addTag`로 연결한 뒤 저장하고, `ProblemResponse.from(saved)`에서 `problem.getTags()`를 쓰므로 **같은 트랜잭션 안에서는** 저장 직후 응답에 태그가 포함되는 구조입니다. 따라서 “표시가 안 된다”는 쪽은 **어디에서 보는지**(등록 직후 vs 목록/태그별 검색)와 **태그별 검색 시 한 개만 보이는 이슈**와 함께 보는 것이 맞습니다.

### 2.2 태그별 검색 시 “해당 태그만 보이고 나머지는 안 보임”
- **원인**: `findDistinctByTags_Id(tagId)`로 생성되는 JPQL은 대략  
  `select distinct p from Problem p join p.tags t where t.id = :tagId`  
  형태입니다. 이때 **join만 하고 fetch는 하지 않으면**, Hibernate가 해당 조인에 참여한 **그 태그만** `Problem`의 `tags` 컬렉션에 넣어 두는 동작을 할 수 있고, lazy load 시점에도 이미 “필터된” 상태가 유지되는 경우가 있습니다.  
  그래서 **태그별 검색 결과**에서는 각 문제에 대해 **검색에 사용한 그 tagId에 해당하는 태그만** `tags`/`tagIds`에 들어가고, 같은 문제에 붙어 있는 **다른 태그들은 응답에 포함되지 않는** 현상이 발생합니다.
- **결과**: “아이디에 같이 있는 이름만 태그로 표시되고 나머지는 안 된다”는 현상과 정확히 일치합니다.

### 2.3 원하시는 동작과의 차이
- **현재**: DB에 없는 이름을 입력해도 `findOrCreateByName`으로 **새 태그가 생성**되고, 문제에 연결됨 → 자유 태깅.
- **원하시는 상태**: **DB에 미리 정의된 태그만** 존재하고, 사용자는 **그 선택지 안에서만** 태그를 골라 쓰는 것.
- 따라서 “설정 가능한 태그를 DB에 두고, 사용자는 그 선택지에서만 사용”하려면 **문제 등록/수정 시에는 tagNames로 새 태그를 만들지 않고, tagIds(또는 허용된 태그 이름 목록)만 받아서 기존 태그에만 연결**하도록 바꿔야 합니다.

## 3. 개선 계획

### 3.1 태그별 검색 시 “모든 태그”가 보이도록 수정 (백엔드)
- **목표**: `GET /api/problems?tagId=...` 응답의 각 문제에, **해당 문제에 붙은 전체 태그**가 `tags`/`tagIds`에 포함되도록 함.
- **방법**  
  - `findDistinctByTags_Id`만 쓰지 말고,  
    “태그로 걸러낸 문제 ID 목록”을 구한 뒤, **해당 문제들을 `tags`를 fetch해서 조회**하는 방식으로 변경  
  - 예:  
    - `@Query("SELECT DISTINCT p FROM Problem p LEFT JOIN FETCH p.tags WHERE p IN (SELECT p2 FROM Problem p2 JOIN p2.tags t WHERE t.id = :tagId)")`  
    - 또는 `findDistinctByTags_Id`로 ID만 받고, `problemRepository.findAllById(..., Pageable)` 등으로 **`@EntityGraph(attributePaths = "tags")`**를 붙인 조회를 한 번 더 하는 방식  
  - 이렇게 하면 태그별 검색 결과에서도 카드에 **모든 태그**가 표시됩니다.

### 3.2 “DB에 설정한 태그만 사용”하도록 정책 전환
- **백엔드**
  - 문제 등록/수정 시 **`tagNames`로 새 태그를 생성하지 않도록** 변경.
  - 선택지:
    - **A**: `tagNames` 제거하고 **`tagIds`만** 받기.  
      - 등록/수정 시 `tagIds`만 사용하고, 목록에 없는 id는 무시하거나 400 처리.
    - **B**: `tagNames`는 유지하되 **“이미 존재하는 태그 이름만”** 허용 (이름으로 조회만 하고, 없으면 무시 또는 400).  
      - 관리자가 미리 태그 이름을 만들어 두면, 사용자는 이름으로도 보내줄 수 있음.
  - 관리용 태그는 **DB에 미리 insert**하거나, **태그 관리 화면/API**(이미 있는 `POST /api/tags`, 목록 조회)로만 추가하도록 안내.
- **프론트엔드**
  - 문제 등록/수정 화면에서 **자유 입력 대신** `GET /api/tags`로 가져온 목록을 기준으로 **다중 선택(드롭다운/체크박스)**만 하도록 변경.
  - 전송 시 **`tagIds`만** 보내거나, 허용된 이름만 쓰는 방식으로 통일.

### 3.3 등록 직후 “태그가 보이는지” 확인
- 생성 API 응답에 이미 `tags`/`tagIds`가 들어가므로, **등록 완료 화면**에서 응답의 `tags`를 그대로 표시해 주면 “등록 시 태그가 안 보인다”는 체감을 줄일 수 있음.
- 동시에 3.1을 적용하면, 사용자가 “태그별 검색”으로 들어가도 같은 문제의 **모든 태그**가 보이므로, 두 현상이 함께 해소됩니다.

### 3.4 작업 순서 제안
1. **3.1** 태그별 검색 시 문제별 전체 태그가 조회되도록 리포지토리/서비스 수정 (즉시 체감 개선).
2. **3.2** 백엔드에서 “기존 태그만 허용” 정책 적용 (`tagNames` 제거 또는 “존재하는 이름만” 허용).
3. **3.2** 프론트: 태그를 선택지 기반으로만 입력하도록 변경 (`tagIds` 위주).
4. **(선택)** 등록 완료 화면에서 응답의 `tags` 표시 (3.3).

## 4. 요약

| 구분 | 현재 | 목표 |
|------|------|------|
| 태그 생성 | 문제 등록/수정 시 이름 입력만으로 자동 생성 | DB/관리 화면에서만 생성, 사용자는 선택지만 사용 |
| 태그별 검색 결과 | 검색한 태그 하나만 각 문제에 표시됨 | 해당 문제의 **모든 태그** 표시 |
| 문제 등록 시 표시 | 응답에 태그는 있으나, 목록/태그별 검색에서 한 개만 보여 체감상 “안 보임” | 선택지 기반 + 태그별 검색 시 전부 표시로 일관되게 보임 |

위 순서대로 적용하면 “DB에 설정한 태그만 선택해서 쓰고, 태그별 검색에서도 모든 태그가 보이는” 상태로 정리할 수 있습니다.
